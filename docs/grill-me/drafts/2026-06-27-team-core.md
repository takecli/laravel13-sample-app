# ドラフト: チーム（team-core）

> 内蔵 grilling で叩いた決定事項を集約した **brainstorming 入力用ドラフト**。
> これは確定 spec ではない。確定は superpowers:brainstorming が行う。
> 1 ファイル = 1 機能。API/UI 等のレイヤーはこの機能の下位章として持つ。
> 本ドラフトは `docs/requirements.md`（正本）の確定事項を転記し、未確定のみ §9 に残した。

## 0. メタ
- 機能名: チーム（team-core）
- 作成日: 2026-06-27
- grilling 実行: 一部（正本 requirements.md からの転記。未確定は §9）
- 関連機能（依存・関連する他ドラフト）:
    - `auth`（MVP・既存維持）: `api/*` は Keycloak セッション必須。作成者 = 認証ユーザー（`users`）。
    - `team-membership`（MVP・後続）: 本機能の「作成者を自動 owner」は `team_members` への 1 行追加を伴うため境界が重なる（§9 で整理）。ロール変更・除名・脱退は team-membership 側。
    - `article-core` ほか `article-*`（MVP/P2）: チームに属し、認可は `team_members` 経由（本機能の後続）。
    - `team-access`（P2）: 参加申請（public）/ 招待（private）。`visibility` の意味づけは本機能で確定。

## 1. 目的（Why）
チームの暗黙知を出し分けて蓄積するため、ナレッジの所属単位となる **チーム** を CRUD で管理できるようにする（正本 §1.3 / §4.3）。

- 社外秘と公開を出し分ける土台として、チームに `visibility`（public/private）を持たせる（正本 §1.2 / §4.3）。
- 後続の記事・メンバー・申請/招待が全てチームに紐づくため、MVP コアの基盤機能（正本 §11 MVP ステップ 3）。
- 既存 `Team` 縦切り（list/create/update のみ）を正本 §5 のデータモデルに合わせて作り直し、取得（単体）・削除（論理削除）・`slug`・`owner_user_id` を備えた完全な CRUD にする。

## 2. スコープ
- やること:
    - チームの **作成 / 取得（単体） / 一覧 / 更新 / 削除（論理削除）**（正本 §4.3 / §9）。
    - `visibility`: `public`（誰でも申請可・記事は閲覧可）/ `private`（招待制・非メンバーは不可）を保持（正本 §4.3 / §10 `Team\Visibility`）。
    - `slug` の一意性（正本 §5 teams 行 `slug`(uniq)）。
    - 作成者を自動で `owner` ロールにする（正本 §4.3）→ `team_members` に owner 行を作成（§9 で境界を確定）。
    - 既存 `teams` テーブルを正本 §5 に合わせて作り直す（命名・enum 統一。`public_status` → `visibility(public/private)`、`slug`・`owner_user_id`・`icon_url` を追加）。
    - 監査列（`created_at/updated_at`、作成/更新/削除ユーザーID）と `deleted_at`（論理削除）を持つ（正本 §5 冒頭の方針）。
- やらないこと（今回対象外）:
    - メンバー一覧 / ロール変更 / 除名 / 脱退（→ `team-membership` §4.4。最後の owner 不在禁止も向こう）。
    - 参加申請 / 招待（→ `team-access` §4.5、P2）。
    - 記事・タグ・添付・コメント等のチーム配下リソース（各 `article-*`）。
    - 通知連携（→ `notifications` §4.15、P3）。
- 非ゴール（誤解されやすいので明示）:
    - `visibility` による記事の可視性制御そのものは記事機能側の責務（本機能は値の保持と整合のみ）。
    - チーム横断の組織オーナー（Owner ペルソナ）専用の管理画面（正本では未規定。本機能では「作成者 = owner ロール」までに限定）。
    - 物理削除・カスケード削除（論理削除のみ。配下記事の扱いは §9）。

## 3. ユースケース / ユーザーストーリー
- [ ] US1: 認証ユーザーは新しいチームを作成でき、自身が自動的に `owner` になる（正本 §4.3）。
- [ ] US2: ユーザーはチーム一覧をページング付きで取得できる（正本 §6 性能：一覧はページング必須）。
- [ ] US3: ユーザーはチーム単体を ID で取得できる（正本 §9 `GET /teams/{id}`）。
- [ ] US4: Owner / Admin はチーム設定（名前・説明・visibility・icon 等）を更新できる（正本 §8 「チーム設定変更・削除」= Owner/Admin）。
- [ ] US5: Owner / Admin はチームを論理削除できる（正本 §8）。
- [ ] US6: `slug` は全チームで一意で、重複作成・重複更新はバリデーションで弾かれる（正本 §5）。

## 4. データモデル設計
- エンティティ: `Team`（ドメイン）/ `teams`（テーブル、正本 §5）。
- 属性 / 型 / 制約（正本 §5 teams 行 + §5 冒頭の共通方針）:
    - `id`: `CHAR(36)` UUID, PK。
    - `name`: `VARCHAR(255)` NOT NULL（チーム名）。
    - `slug`: `VARCHAR` NOT NULL, **UNIQUE**（URL 用識別子）。
    - `description`: `TEXT`（説明）。
    - `visibility`: `ENUM('public','private')` NOT NULL（正本 §10 `Team\Visibility`）。
    - `icon_url`: チームアイコン URL（任意）。
    - `owner_user_id`: `CHAR(36)`（作成者 = オーナー、`users.id` 参照）。
    - 監査: `created_at` / `updated_at`、`created_user_id` / `updated_user_id` / `deleted_user_id`。
    - `deleted_at`: `TIMESTAMP NULL`（論理削除）。
- リレーション（正本 §5.2）:
    - `users 1─* team_members *─1 teams`（所属・ロール。owner 行は本機能の作成時に 1 件作る）。
    - `teams 1─* articles`、`teams 1─* team_join_requests / team_invitations`、`teams 1─* collections`（いずれも後続機能）。
    - `teams.owner_user_id ─ users.id`（作成者）。
- enum:
    - `App\Domains\Enums\Team\Visibility: string { Public='public'; Private='private'; }`（正本 §10）。既存 `PublicStatus`（invitation/public）は正本に存在しないため `Visibility` に置き換え（作り直し前提）。
    - 作成時に付与するロールは `App\Domains\Enums\Team\MemberRole::Owner`（正本 §10。実体定義は team-membership 側だが、本機能の owner 行作成で参照）。

## 5. API 設計
prefix `/api/v1`、Keycloak セッション必須、`ApiResponse` エンベロープ（`data/message/result`）準拠（正本 §6 / §9）。

| メソッド | パス | 概要 | 主なリクエスト | 主なレスポンス | エラー |
|---|---|---|---|---|---|
| GET | `/teams` | チーム一覧（ページング・絞り込み） | query: `name?`, `visibility?`, `page?`, `limit?`, `sort?` | `data`: チーム配列＋ページング | 401（未認証）/ 422（不正パラメータ）/ 500 |
| POST | `/teams` | チーム作成（作成者を owner に） | body: `name`(必須), `slug`(必須・uniq), `description?`, `visibility`(必須), `icon_url?` | `data`: 作成したチーム | 401 / 422（必須・enum・slug 重複）/ 500 |
| GET | `/teams/{id}` | チーム単体取得 | path: `id` | `data`: チーム | 401 / 404（不存在/論理削除済み）/ 500 |
| PUT | `/teams/{id}` | チーム更新 | path: `id` / body: `name?`, `slug?`(uniq), `description?`, `visibility?`, `icon_url?` | `data`: 更新後チーム | 401 / 403（Owner/Admin 以外）/ 404 / 422（slug 重複等）/ 500 |
| DELETE | `/teams/{id}` | チーム論理削除 | path: `id` | `data`: なし or 削除済み ID | 401 / 403 / 404 / 500 |

- バリデーションは FormRequest、`toInput()` で Application 入力 DTO に詰め替え（`.claude/rules/php.md`、既存 `CreateTeamRequest` 準拠）。
- ルートは `routes/web_api.php` の `teams` グループ（`middleware => auth`）に GET単体 / DELETE を追加し、`{team_id}` パラメータ名は既存に合わせる（正本 §9 のパスは `{id}` 表記だが既存実装は `{team_id}`。§9 未確定）。
- 例外は `try/catch (Exception $e)` → `Log::error(__('messages.error', ...))` + `report($e)` + `ApiResponse::serverError()`（既存 `TeamController` 準拠）。作成/更新/削除は `DB::transaction`（owner 行作成を含むため作成は必須）。

## 6. UI / 画面設計

## 7. 技術選定と根拠
- 採用:
    - 既存 `Team` 縦切り（Http→Applications→Domains→Infra）を踏襲し、取得単体・削除・slug・owner を追加（正本 §6 / `CLAUDE.md`）。
    - `slug` 一意は DB UNIQUE 制約 ＋ FormRequest の `unique` ルール（二重防御）。
    - 削除は `deleted_at` による論理削除（正本 §5 冒頭方針）。
    - enum `Team\Visibility`（バック付き）で区分値表現（正本 §10）。
    - 作成は `DB::transaction` で teams 行 + team_members(owner) 行を原子的に作る。
- 理由:
    - 正本が「既存マイグレーションは本モデルに作り直してよい」と明記（正本 §0 / §5）。命名・enum を正本へ統一できる。
    - リファレンス踏襲で依存方向・テスト方針・カバレッジ 100% を満たしやすい。
- 却下した代替案 / 却下理由:
    - 既存 `public_status(invitation/public)` を流用 → 正本 §10 の `visibility(public/private)` と不一致。作り直しが正本方針。
    - 物理削除 → 監査・復元要件（論理削除方針）に反する。
    - owner 行作成を team-membership に丸投げ → 「作成者は自動 owner」は team-core の受入条件（正本 §4.3）。作成トランザクション内で行う必要（境界は §9 で確定）。

## 8. エッジケース・非機能
- エッジケース:
    - `slug` 重複（作成・更新）→ 422。大文字小文字・前後空白の正規化方針は §9。
    - 論理削除済みチームの取得 / 更新 / 削除 → 404。
    - 同名 `name` は許容（一意は `slug` のみ）。
    - 削除時に配下に記事・メンバーが存在する場合の扱い（カスケード or ブロック）→ §9。
- 認可 / 権限（正本 §8）:
    - 一覧・取得: 認証ユーザー（Guest=未所属認証ユーザーも一覧/公開閲覧可否は §9）。
    - 作成: 認証ユーザーなら可（作成者が owner になる）。
    - 更新・削除: **Owner / Admin のみ**（`team_members` のロール参照）→ Member/Guest は 403。
    - 認可判定に `team_members` ロールが必要なため team-membership とロジック共有（§9）。
- 失敗時の挙動: 例外は `Log::error` + `report()` + `ApiResponse::serverError()`。作成/更新/削除は失敗時 `DB::rollBack()`（既存 `TeamController` 準拠）。
- 性能・可用性で意識する点: 一覧はページング必須（既定 `Pagination`）。一覧で N+1 を避ける（owner/メンバー数を出すなら Infra で eager load）。`slug` に UNIQUE インデックス。

## 8.5 テスト方針（準拠先 + 機能固有の観点）
- 準拠 rules:
    - バックエンド（該当する場合）: `.claude/rules/php.md`（PHPUnit 12、`#[Test]` 属性、AAA、Mockery、行カバレッジ 100%、Feature は MySQL テスト DB `learning_portal_testing`）。
    - フロントエンド（該当する場合）: `.claude/rules/typescript.md`（`tsc --noEmit` / ビルド可、`api/team.ts` 経由、`ApiError` 捕捉）。
- 本機能固有のテスト観点:
    - 到達が難しい分岐 / 要モックの依存:
        - Controller の catch（作成/更新/削除/取得/一覧）: `TeamRepositoryInterface` を `$this->app->instance(...)` で `andThrow` モックに差し替え、`serverError` 到達と `DB::rollBack` を検証。
        - 認可分岐（Owner/Admin vs Member/Guest = 403）: ロール別にケースを用意。
        - 作成トランザクション内で owner 行作成が失敗した場合のロールバック。
    - 重点的に検証したいユースケース（§3 と対応）:
        - US1: 作成時に teams 行と team_members(owner) 行が両方作られる（Feature・実 DB）。
        - US6: `slug` 重複が 422（作成・更新の両方）。
        - US3/US5: 論理削除済みチームの取得・再削除が 404。
        - 一覧のページング・visibility 絞り込み。
    - DB / 外部サービス依存で注意する点:
        - マイグレーションは MySQL 専用の生 SQL（`ENUM`/`COMMENT`/UUID）。sqlite では動かないため Feature は MySQL テスト DB 前提。
        - UNIQUE(`slug`) 制約と FormRequest `unique` の二重防御を両方テスト（DB レベルと検証レベル）。
