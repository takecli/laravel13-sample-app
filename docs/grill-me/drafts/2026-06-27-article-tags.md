## 0. メタ
- 機能名: タグ（article-tags）
- 作成日: 2026-06-27
- grilling 実行: 一部（正本 requirements.md からの転記。未確定は §9）
- 関連機能（依存・関連する他ドラフト）:
    - 依存: `article-core`（§4.6 記事。タグ付け対象・一覧のタグ絞り込みの母体）
    - 認可の前提: `team-membership`（§11 依存の要点「`article-*` は `team-membership`（認可）に依存」）
    - 関連: `search`（§4.14、P2。キーワード＋タグの全文検索。MVP のタグ絞り込みとは別機能）

## 1. 目的（Why）
記事にタグを付与して分類・横断的な発見性を高める（§4.8）。タグの正規化・重複防止により表記ゆれを抑え、記事一覧の **タグ絞り込み**（§4.6「チーム・状態・タグ・著者・キーワードで絞り込み」、US-R1）を成立させる。

## 2. スコープ
- やること:
    - タグの作成（正規化・重複防止）と一覧取得（`GET /tags`・`POST /tags`、§9 API）。
    - 記事へのタグ付け / 付け替え（`PUT /articles/{id}/tags`、§9 API。差分更新ではなく「その記事のタグ集合を丸ごと置換」する想定）。
    - 記事一覧でのタグ絞り込み（母体は `article-core` の `GET /articles`。本機能はタグ条件の提供側）。
    - データモデル: `tags`（`name` uniq / `slug` uniq）と中間テーブル `article_tags`（`article_id`×`tag_id` 複合 uniq）（§5.1）。
- やらないこと（今回対象外）:
    - 全文検索（`search`〔P2〕）。MVP のタグ絞り込みは `articles` 一覧の `LIKE`/結合フィルタで足り、Meilisearch 等の検索基盤は対象外（§4.14）。
    - タグの改名・削除・マージ（正本に明記なし。§9 で要確認）。
    - タグの階層・色・説明等の付加メタ（`tags` の列は `name`/`slug` のみ、§5.1。YAGNI §1.4）。
- 非ゴール（誤解されやすいので明示）:
    - タグは記事に対してのみ付く。チーム/ユーザー/コレクション等へのタグ付けは対象外。
    - タグはグローバル（全社共有のマスタ）であり、チームごとのタグ名前空間ではない（`tags.name` がグローバル uniq、§5.1）。

## 3. ユースケース / ユーザーストーリー
- [ ] US1: Contributor（記事作成権限のあるメンバー）が記事作成・編集時にタグを付け、必要に応じて付け替えできる（US-A1「添付・タグを付け…公開できる」）。
- [ ] US2: Learner が記事一覧をタグで絞り込んで目的の記事を見つけられる（US-R1「タグで絞り込める」）。
- [ ] US3: ユーザーがタグを新規作成すると、既存と表記が一致するタグは重複作成されず既存に正規化される（§4.8「正規化・重複防止」）。
- [ ] US4: ユーザーが利用可能なタグの一覧を取得し、付与候補として参照できる（`GET /tags`）。

## 4. データモデル設計
- エンティティ:
    - `tags`（タグマスタ。グローバル）
    - `article_tags`（記事×タグの中間テーブル）
- 属性 / 型 / 制約（§5.1、§5「全テーブル `id` は `CHAR(36)` UUID、監査列・論理削除方針を踏襲」）:
    - `tags.id`: `CHAR(36)` UUID（PK）
    - `tags.name`: 文字列 / **uniq**（表示名。正規化後にユニーク判定）
    - `tags.slug`: 文字列 / **uniq**（URL/絞り込みキー）
    - 監査列: `created_at` / `updated_at`、作成/更新/削除ユーザーID、論理削除対象なら `deleted_at`（§5 既存方針）。※ tags に論理削除を持たせるかは §9。
    - `article_tags.id`: `CHAR(36)` UUID（PK）
    - `article_tags.article_id`: FK → `articles.id`
    - `article_tags.tag_id`: FK → `tags.id`
    - `article_tags` 複合 uniq: (`article_id`, `tag_id`)（同一記事への同一タグ二重付与を防止、§5.1）
- リレーション:
    - `articles *─* tags`（`article_tags` 中間、§5.2）。
    - enum: タグ機能に区分値 enum は無し（§10 にタグ enum の定義なし）。

## 5. API 設計
`prefix: /api/v1`、Keycloak セッション必須、`ApiResponse` エンベロープ準拠（§9 前文）。

| メソッド | パス | 概要 | 主なリクエスト | 主なレスポンス | エラー |
|---|---|---|---|---|---|
| GET | `/tags` | タグ一覧取得（付与候補・絞り込み候補） | クエリ（名前部分一致・ページング想定。§6 一覧はページング必須） | `tags[]`（`id`/`name`/`slug`） | 401（未認証） |
| POST | `/tags` | タグ作成（正規化・重複防止） | `name`（必須） | 作成/既存の `tag`（`id`/`name`/`slug`） | 401 / 422（検証）/ 重複時の扱いは §9 |
| PUT | `/articles/{id}/tags` | 記事のタグ集合を置換（付け替え） | `tag_ids[]` または `names[]`（指定方式は §9） | 更新後の `tags[]`（その記事の現タグ） | 401 / 403（記事編集権限なし）/ 404（記事なし）/ 422 |

- レスポンスは `{ data, message, result }`（§6 可観測性、CLAUDE.md）。一覧 API はページング既定（`Pagination`、§6 性能）。

## 6. UI / 画面設計
> 記事 UI（`article-core`）に内包される。タグ単独の画面は持たない想定。
- 画面 / 状態:
    - 記事作成/編集フォーム内の **タグ入力**（既存タグのサジェスト＋新規作成。`GET /tags` 候補、`POST /tags` で新規）。
    - 記事一覧の **タグフィルタ**（選択タグで `GET /articles?tag=...` を呼ぶ。タグ条件は `article-core` 一覧 UI の一部）。
    - 記事詳細での **タグ表示**（クリックで該当タグ絞り込みへ）。
- 主要操作: タグ選択・追加（新規作成含む）・除去 → 保存で `PUT /articles/{id}/tags`。
- 入力バリデーション・エラー表示: 空タグ・長さ上限・無効文字は入力時に弾く（具体ルールは §9）。API 422 は `ApiError` 捕捉でフォームにフィードバック（typescript.md）。

## 7. 技術選定と根拠
- 採用:
    - 正規化済み `tags` マスタ ＋ 中間テーブル `article_tags`（多対多）。
    - タグ付け替えは **集合置換**（`PUT /articles/{id}/tags` で現タグを全置換）。
    - MVP のタグ絞り込みは `articles` 一覧クエリ側の結合フィルタで実装（検索基盤に依存しない）。
- 理由:
    - §5.1 が `tags`/`article_tags` を独立テーブルかつ uniq 制約付きで定義済み。正規化マスタにより表記ゆれと重複を構造的に防げる（§4.8）。
    - 集合置換は API が冪等で実装・テストが単純（差分 add/remove より分岐が少ない）。
    - `search`（P2）が来るまで全文検索基盤を導入しない（§1.4 YAGNI、§4.14）。
- 却下した代替案 / 却下理由:
    - 記事行にカンマ区切りタグ文字列を保持: 正規化・uniq・絞り込み索引が破綻するため却下（§5.1 のテーブル設計に反する）。
    - チームごとのタグ名前空間: `tags.name` がグローバル uniq と定義済み（§5.1）。複雑化のため却下。
    - 付け替えを add/remove 差分 API にする: 冪等性・テスト容易性で集合置換を優先。

## 8. エッジケース・非機能
- エッジケース:
    - `POST /tags` で正規化後に既存タグと一致 → 新規作成せず既存タグを返す（重複防止、§4.8）。HTTP ステータス（200 既存返却 か 409 か）は §9。
    - `PUT /articles/{id}/tags` に空配列 → 全タグ解除。
    - 同一タグの重複指定 → `article_tags` 複合 uniq で一意化（重複は無視）。
    - 存在しない `tag_id` 指定 → 422 で弾く（あるいは name 指定方式なら未存在は作成、§9）。
    - 非 ASCII（日本語）タグ名の slug 生成 → 衝突・空 slug の懸念（§9）。
- 認可 / 権限（§8）:
    - タグ付け/付け替えは「記事の作成・自分の記事編集」権限に従う（Owner/Admin/Member ✓、Guest ✗）。他者記事へのタグ編集は Owner/Admin のみ（§8「他者記事の編集・削除」）。
    - タグ一覧 `GET /tags`・タグ作成 `POST /tags` の必要ロールは §8 マトリクスに明記なし（§9）。
    - 全 API で Keycloak セッション必須（§6 認可）。
- 失敗時の挙動: 例外は `Log::error` ＋ `report()` ＋ `ApiResponse::serverError()` の定型（§6 可観測性、php.md）。
- 性能・可用性で意識する点:
    - `GET /tags`・タグ絞り込みはページング必須（§6 性能）。
    - 記事一覧でタグ結合する際 N+1 を作らない（Infra で eager load、§6 性能）。
    - `tags.name`/`tags.slug`/`article_tags` 複合キーにインデックス（uniq 制約由来）。

## 8.5 テスト方針（準拠先 + 機能固有の観点）
- 準拠 rules:
    - バックエンド（該当する場合）: `.claude/rules/php.md`
    - フロントエンド（該当する場合）: `.claude/rules/typescript.md`
- 本機能固有のテスト観点:
    - 到達が難しい分岐 / 要モックの依存:
        - `POST /tags` の「正規化後に既存ヒット → 既存返却」分岐と「新規作成」分岐の双方（Repository インターフェースをモックして UseCase で両分岐到達）。
        - 例外定型（catch → `serverError`）は Repository モックの `andThrow` で到達させる。
        - 認可分岐（自記事 / 他者記事 / Guest）は §8 マトリクス通りに到達させる。
    - 重点的に検証したいユースケース（§3 と対応）:
        - US3 正規化・重複防止（同義タグが二重作成されない、`tags.name`/`slug` uniq）。
        - US1 付け替え＝集合置換（追加・解除・空配列）の冪等性と `article_tags` 複合 uniq。
        - US2 記事一覧のタグ絞り込み（`article-core` 一覧と結合した Feature テスト）。
    - DB / 外部サービス依存で注意する点:
        - Feature/Repository テストは MySQL テスト DB 前提（`learning_portal_testing` / `.env.testing`）。マイグレーションは MySQL 専用 SQL（`UUID_TO_BIN` 等）で sqlite 不可（php.md）。
        - `tags`/`article_tags` の uniq・複合 uniq 制約違反は DB レイヤで起きるため、Repository の Feature テストで実 DB を使って検証する。
