## 0. メタ
- 機能名: 認証 / アカウント（auth）
- 作成日: 2026-06-27
- grilling 実行: 一部（正本 requirements.md からの転記。未確定は §9）
- 関連機能（依存・関連する他ドラフト）:
    - `user-profile`（§4.2）: `users` レコードの表示名・アバター・自己紹介・貢献度を扱う。auth は `users` の生成元（upsert）であり、プロフィール列の確定はこちらに依存する。
    - 全 MVP 機能（§6・§9）: `api/*` は本機能が確立する Keycloak セッション必須。auth は認可境界の前提となる。

## 1. 目的（Why）
- Keycloak（OIDC / Socialite・セッション/BFF 方式）による社内 SSO ログインを提供し、`api/*` の認可境界の前提を確立する（正本 §4.1・§8・§13）。
- 認証は **Keycloak / BFF のみ**。メール＋パスワード等の他方式は持たない（非ゴール §13）。
- 既存実装を**維持**しつつ、正本との差分（`users` の `keycloak_sub` upsert、ログアウト）を埋める（§4.1・§11 MVP-1「既存維持・`users` upsert 追加」）。

## 2. スコープ
- やること:
    - Keycloak OIDC ログイン: `redirect`（認可エンドポイントへ誘導）→ `callback`（コード交換・ユーザー情報取得）。セッション/BFF 方式（サーバセッションに保持、フロントは Cookie）。
    - **初回ログイン時に `users` を `keycloak_sub` で upsert**（名寄せキー）。既存ユーザーは同 sub で同定。
    - 認証ユーザー情報の取得（`GET /auth`）。
    - ログアウト（`POST /auth/logout`）。
- やらないこと（今回対象外）:
    - プロフィール編集・他ユーザー公開プロフィール（→ `user-profile` §4.2）。
    - ロール×権限の判定ロジック本体（→ `team-membership` §4.4・§8。auth は「認証済みか」までを担保）。
- 非ゴール（誤解されやすいので明示, §13）:
    - Keycloak 以外の認証方式（メール/パスワード、他 IdP）。
    - 独自のユーザー登録・パスワード管理画面（ユーザーは Keycloak 側で管理）。

## 3. ユースケース / ユーザーストーリー
- [ ] US1: 未認証ユーザーがログイン導線を踏むと Keycloak の認可画面へリダイレクトされる（`GET /auth/keycloak/redirect`）。
- [ ] US2: Keycloak 認証後、callback でアプリにセッションが確立し、アプリ画面へ戻る（`GET /auth/keycloak/callback`）。
- [ ] US3: **初回**ログインのユーザーは `users` に `keycloak_sub` 起点でレコードが作成される。2 回目以降は同一レコードに名寄せされる（重複作成しない）。
- [ ] US4: 認証済みユーザーは自分の認証情報を取得できる（`GET /auth`）。未認証時はサーバが `data: null` を返す（フロントは未認証として扱う）。
- [ ] US5: ユーザーはログアウトでき、以降 `api/*` は未認証として扱われる（`POST /auth/logout`）。

## 4. データモデル設計
- エンティティ: `users`（利用者 / Keycloak 連携。正本 §5.1）。auth が**生成・名寄せの責任を持つ唯一の機能**。
- 属性 / 型 / 制約（正本 §5.1・§5 冒頭の共通方針）:
    - `id`: `CHAR(36)` UUID（`HasUuids`、`incrementing=false`）。
    - `keycloak_sub`: 文字列・**unique**。Keycloak の `sub` を名寄せキーとする（upsert のキー）。
    - `email`: 文字列。Keycloak から取得。
    - `display_name`: 表示名（正本 §5.1。詳細は `user-profile` で確定）。
    - `avatar_url` / `bio` / `contribution_points`: プロフィール系列（auth では生成時に既定値、編集は `user-profile`）。
    - 監査列: `created_at` / `updated_at`（既存方針）。論理削除対象列の有無は §9 で確認。
- リレーション（参考, §5.2）: `users 1─* team_members` / `articles(author)` / `notifications` / `user_badges` 等。auth 自身は他テーブルへの書き込みを持たない。

> ⚠ 現状コード（`app/Models/User.php`）は `keycloak_id` / `name` を使用し、callback は `User::firstOrCreate(['keycloak_id'=>...])`。正本 §5 は `keycloak_sub` / `display_name`。命名差・列リネーム要否は §9 未確定。

## 5. API 設計
| メソッド | パス | 概要 | 主なリクエスト | 主なレスポンス | エラー |
|---|---|---|---|---|---|
| GET | `/api/v1/auth/keycloak/redirect` | Keycloak 認可画面へリダイレクト | なし | 302 Redirect（Keycloak へ） | 失敗時 `ApiResponse::serverError`（`Log::error`＋`report`） |
| GET | `/api/v1/auth/keycloak/callback` | コード交換・ユーザー情報取得・`users` upsert・セッション確立 | Keycloak からの `code`/`state` | 302 Redirect（アプリ画面へ） | 失敗時はエラー画面へリダイレクト |
| GET | `/api/v1/auth` | 認証ユーザー情報取得 | セッション Cookie | `ApiResponse::success`（`{id,email,display_name,...}`） / 未認証は `data: null` | 例外時 `serverError` |
| POST | `/api/v1/auth/logout` | ログアウト（セッション破棄） | セッション Cookie・CSRF | `ApiResponse::success`（または 204） | 例外時 `serverError` |

- ルート定義は `routes/web_api.php`（`prefix: api/v{version}`）。auth 系は**セッション必須の対象外**（`api/*` 認証必須からは除外, 正本 §9 注記）。
- レスポンスは `ApiResponse` エンベロープ（`{data,message,result}`）統一（§6 可観測性）。文言は `lang/en/*.php` 翻訳キー（§6 国際化）。

> ⚠ 現状: `POST /auth/logout` は**未実装**（ルート無し）。callback の成功リダイレクト先は `/dashbord`（既存タイポ）、失敗時 `/error`。`getUser(... $vesrion)` の引数名タイポは既存に合わせる（CLAUDE.md）。

## 6. UI / 画面設計
- 画面 / 状態:
    - 未認証: ログイン導線（Keycloak redirect への遷移）。フロントは `auth-context` の `loading → user|null` で出し分け。
    - 認証済み: `_authenticated` 配下ルートを表示。`isAuthenticated` で `_unauthenticated` / `_authenticated` を切り替え。
- 主要操作:
    - ログイン: redirect エンドポイントへ遷移 → callback 後にアプリへ戻り `AuthProvider` が `refresh()` で `GET /auth` を再取得。
    - ログアウト: `POST /auth/logout` 呼び出し後に `refresh()` で `user=null` 化。
- 入力バリデーション・エラー表示:
    - フォーム入力は持たない（資格情報は Keycloak 側）。`GET /auth` 失敗（401 等）は `auth-context` が catch して未認証扱い（既存挙動）。
    - 既存実装: `resources/js/api/auth.ts`（`AuthAPI.getCurrentUser`）, `resources/js/context/auth-context.tsx`。ログアウト用 API は未実装（追加要）。

## 7. 技術選定と根拠
- 採用: Keycloak（OIDC）＋ Laravel Socialite（`keycloak` ドライバ）＋ **サーバセッション / BFF**（フロントは Cookie・`credentials: include`）。
- 理由: 社内 SSO を単一 IdP に集約。トークンをブラウザに露出させない BFF でセキュリティを担保（§6 セキュリティ・§13 で他方式を除外）。既存実装の方式をそのまま維持（§4.1「既存を維持」）。
- 却下した代替案 / 却下理由:
    - メール＋パスワード／他 IdP: 非ゴール（§13）。
    - フロントで OIDC トークンを直接保持（SPA + bearer）: トークン露出・XSS リスク。BFF を選択。

## 8. エッジケース・非機能
- エッジケース:
    - callback で Keycloak からユーザー情報が取れない / 例外: エラー画面へリダイレクト（既存）。
    - 同一 `keycloak_sub` での再ログイン: 新規作成せず既存レコードに名寄せ（upsert）。
    - 未認証で `GET /auth`: `data: null`（200 か 401 かは §9）。
    - Keycloak 側のメール変更: 次回ログイン時に `users.email` を更新するか（upsert の update 範囲）→ §9。
- 認可 / 権限（§8）: auth は「認証済みか」までを担保。`api/*`（teams 等）は `auth` ミドルウェアでセッション必須。ロール×権限判定は team 系機能の責務。
- 失敗時の挙動: 例外は `Log::error` ＋ `report()` ＋ `ApiResponse::serverError()`（API 系）／エラー画面リダイレクト（callback）。`AuthController`・`KeycloakController` が既存パターン。
- 性能・可用性で意識する点: Keycloak 障害時はログイン不可（外部依存）。`GET /auth` は軽量（セッションのユーザー 1 件）。

## 8.5 テスト方針（準拠先 + 機能固有の観点）
- 準拠 rules:
    - バックエンド（該当する場合）: `.claude/rules/php.md`
    - フロントエンド（該当する場合）: `.claude/rules/typescript.md`
- 本機能固有のテスト観点:
    - 到達が難しい分岐 / 要モックの依存:
        - `Socialite::driver('keycloak')` を `Socialite::shouldReceive(...)` でモック（redirect / callback の正常系・例外系）。
        - `Auth`（`Auth::user()` / `Auth::login()` / ログアウト）を `Auth::shouldReceive(...)` または実セッションで検証。
        - controller の catch 到達は依存を `andThrow` させる。
    - 重点的に検証したいユースケース（§3 と対応）:
        - US3: **初回 upsert で新規作成・2 回目は名寄せ（重複作成なし）**を `users` 件数で検証（Feature + MySQL テスト DB）。
        - US4: 認証済み→ユーザー情報返却 / 未認証→`data: null`。
        - US5: ログアウト後に `api/*` が未認証になる。
    - DB / 外部サービス依存で注意する点:
        - Keycloak は外部。テストでは Socialite をモックし実通信しない。
        - upsert は実 DB（MySQL テスト DB `learning_portal_testing`）で確認（マイグレーションは MySQL 専用生 SQL）。
