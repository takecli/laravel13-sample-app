## 0. メタ
- 機能名: 検索 〔`search`〕
- 作成日: 2026-06-27
- grilling 実行: 一部（正本 requirements.md からの転記。未確定は §9）
- 関連機能（依存・関連する他ドラフト）:

    - **位置づけ**: リリースは **P2**（§11 Phase 2 の 13 番目）。`article-core` の一覧フィルタ（§4.6）を、横断キーワード全文検索として独立機能に昇格させたもの。
    - **将来の差し替え**: MVP は MySQL（`LIKE`/全文索引）。将来 Meilisearch 等へ差し替え可能なよう **検索ポート（`Domains/Services/〜Interface`）＋ Infra 実装**で抽象化（§4.14, §6）。

## 1. 目的（Why）
散在するナレッジを「探せる・再利用できる」状態にする（§1.2, §1.3-3）。Learner が記事を**全文検索**し、タグ等で絞り込んで目的の知識へ最短到達できるようにする（US-R1）。`article-core` の一覧フィルタが「チーム内の絞り込み」だったのに対し、本機能は**横断キーワード検索**を担い、タイトル・本文・タグを対象に検索する。検索結果にも §8 の認可（visibility／所属）を必ず適用し、見えてはならない記事を結果から除外する。実装方式（MySQL）をポートで隠蔽し、将来の検索エンジン差し替えに備える（§4.14, §6 保守性）。

## 2. スコープ
- やること:
    - 横断キーワード検索 API（`GET /search?q=&tag=&team=&status=`、§9 API）。対象は **タイトル・本文・タグ**（§4.14）。
    - フィルタ: `tag`（タグ）/ `team`（チーム）/ `status`（記事状態）でのクエリ絞り込み。
    - 結果へ **認可（visibility／所属）フィルタを適用**（§8）。検索実行者のロール×記事の `status`×`visibility` で可視範囲を決定。
    - ページング（既定 `Pagination`、§6 性能）。
    - 検索方式を **検索ポート（`Domains/Services/〜Interface`）** で抽象化し、Infra に **MySQL 実装**（`LIKE`/全文索引）を置く（§4.14, §6）。
- やらないこと（今回対象外）:
    - Meilisearch / Elasticsearch 等の**外部検索エンジン導入**（ポートの差し替え先として将来。MVP/P2 は MySQL のみ。§4.14）。
    - 記事以外（コメント・チーム・ユーザー・コレクション等）の横断検索（本機能は記事のみ）。
    - 検索インデックスの非同期再構築・サジェスト/オートコンプリート・検索履歴の保存。
    - 記事 CRUD・タグ CRUD・状態遷移（→ `article-core` §4.6 / `article-tags` §4.8）。
- 非ゴール（誤解されやすいので明示）:
    - AI による意味検索・自動要約・関連度の機械学習（§13 非ゴール）。
    - 日本語の高精度形態素解析を前提とした検索品質保証（MVP は MySQL の範囲。トークナイズ方針は §9 未確定）。
    - `article-core` の一覧 API（`GET /articles`）の置き換え。一覧はチーム内文脈の絞り込みとして併存する。

## 3. ユースケース / ユーザーストーリー
- [ ] US1（US-R1）: Learner はキーワード `q` を入力し、タイトル・本文・タグにマッチする**閲覧可能な**記事一覧を取得できる。
- [ ] US2（US-R1）: Learner は `tag` でマッチ記事を絞り込める。
- [ ] US3: Learner は `team` で特定チームの記事に絞り込める（その記事を閲覧する権限がある範囲で）。
- [ ] US4: Learner は `status` で記事状態を絞り込める（ただし可視範囲外の状態は強制的に除外＝§8）。
- [ ] US5（§8 認可）: 検索結果には、実行者がアクセスできない記事（非所属 private チームの `team` 記事、`draft`/`in_review`/`archived` で著者・Admin 以外、等）は**含まれない**。
- [ ] US6: 結果はページングされ、0 件時は空配列＋ページング情報を返す。
- [ ] US7（保守性）: 検索実装は MySQL だが、UseCase は検索ポート経由で呼ぶため、将来 Meilisearch 実装へバインドを差し替えても UseCase/Controller は無改修。

## 4. データモデル設計
> 本機能は**新規テーブルを持たない**（検索は既存データの読み取り）。対象は `article-core`/`article-tags` のテーブル。

- エンティティ（参照のみ・§5.1）:
    - `articles`: `title`, `body`(MD), `status`(`Article\Status`), `visibility`(`Article\Visibility`), `team_id`, `author_user_id`, `published_at`, `deleted_at`。
    - `tags` / `article_tags`: タグ絞り込み・タグ語のマッチ対象。
    - `teams`（`visibility`）/ `team_members`（実行者の所属・ロール）: 認可判定に参照。
- ドメインモデル（本機能で新設）:
    - `SearchArticlesSearch`（Filter）: `keyword`（`q`）, `tag`, `teamId`(`team`), `status`(`Article\Status`), ＋認可コンテキスト（実行者 user_id・所属チーム集合）, ページング（`page`/`limit`/`sort`）。
    - `SearchArticlesResult`（Result）: マッチ記事の配列＋総件数等のページング情報。スコア/並び順の保持可否は §9 未確定。
    - 検索ポート: `Domains/Services/ArticleSearchInterface`（能力で命名。ベンダー名を出さない＝`php.md`）。メソッド IF は §9 未確定。
- 属性 / 型 / 制約: 新規列なし。検索性能のための索引は §8 で記述（`article-core` のマイグレーションへ全文索引/フィルタ列索引を持たせる前提）。
- リレーション（§5.2 既存）: `articles *─* tags`（`article_tags` 中間）、`articles *─1 teams`、`teams 1─* team_members`。

## 5. API 設計
`prefix: /api/v1`、Keycloak セッション必須。`ApiResponse` エンベロープ（`data/message/result`）準拠（§9, §6）。

| メソッド | パス | 概要 | 主なリクエスト | 主なレスポンス | エラー |
|---|---|---|---|---|---|
| GET | `/search` | 記事の横断全文検索（認可フィルタ適用） | クエリ: `q`(keyword), `tag`, `team`, `status`, `page`, `limit`, `sort` | `data`: マッチ記事配列＋ページング情報 | 401（未認証）/ 422（不正なフィルタ値・不正 `status`/`team`）/ 500 |

- バリデーションは FormRequest（§6 セキュリティ）。検証済み入力は `toInput()` で Application 層 Input DTO（`SearchArticlesInput`）へ詰め替え（`php.md`）。
- enum 変換（`status` 文字列→`Article\Status`）は FormRequest 側で実施。検証後の値のみ扱う（`tryFrom`）。
- 認可コンテキスト（実行者の所属・ロール）は Controller/UseCase で組み立て、Filter に載せて Infra へ渡す（結果除外を**クエリレベル**で行う方針か後段フィルタかは §9 未確定）。
- `q` 未指定時の挙動（全件＝フィルタのみ検索か 422 か）は §9 未確定。

## 6. UI / 画面設計
> フロントは `resources/js`（`typescript.md` 準拠、`client` 経由・`api/search.ts` に集約）。

- 画面 / 状態:
    - 検索画面: 検索ボックス（`q`）＋フィルタ（タグ／チーム／状態）、結果リスト、ローディング／空（0 件）／エラー状態。
    - 結果アイテム: タイトル（記事詳細へ遷移）、所属チーム、状態バッジ、タグ、抜粋（ハイライト有無は §9）。
- 主要操作:
    - キーワード入力 → 検索実行（送信トリガ：Enter／ボタン。デバウンス有無は §9）。
    - フィルタ選択（タグ／チーム／状態）→ 再検索。ページング操作。
- 入力バリデーション・エラー表示:
    - 空クエリ時の扱いはサーバ仕様に追従（§9）。サーバ 422 を `ApiError` で受けてフィールド/全体メッセージへ。
    - 401 はログイン導線へ。エラーは握り潰さない（`typescript.md`）。

## 7. 技術選定と根拠
- 採用:
    - **検索ポート抽象化**: `Domains/Services/ArticleSearchInterface`（ポート）＋ `Infra` 実装（MySQL）。バインドは `AppServiceProvider`（`$this->app->bind(ArticleSearchInterface::class, MySql実装::class)`）。`FileStorageInterface`＋`Infra/External/Aws/S3` と同じポート/アダプタ流儀（§6, `php.md`）。
    - **MySQL 全文検索/`LIKE`**: MVP/P2 はインフラ追加なしで成立（§4.14）。実装の具象（`MATCH ... AGAINST` / `LIKE`・索引）は Infra 層に閉じる。
    - レイヤード縦切り: Input/Output/UseCase/Search(Filter)/Result/Controller/Request/Resource（`Team` 参照実装踏襲、§6）。
    - 認可フィルタは Domain の Filter（認可コンテキスト）＋ Infra のクエリ条件で表現し、可視範囲外を結果に含めない。
- 理由:
    - 検索実装は将来差し替え確度が高い（§4.14 が明示）。ポートで隔離すれば UseCase/Controller/テストを無改修で乗せ替えられる。
    - 記事の可視性ロジックは `article-core`/§8 に既にあるため、検索でも**同一の認可規則**を再利用し二重定義を避ける（規則の単一情報源化は §9 で要整理）。
- 却下した代替案 / 却下理由:
    - MVP からの外部検索エンジン（Meilisearch/Elasticsearch）導入 → YAGNI（§4.14 は MySQL で抽象化のみ／§1.4）。運用コスト・依存増。ポートだけ用意して後送り。
    - 取得後にアプリ側で全件を認可フィルタ → 大量データで性能劣化・ページング不整合。可能な限りクエリで絞る（方針確定は §9）。
    - `article-core` の `GET /articles` に検索を相乗り → 横断検索（チーム跨ぎ）と一覧（チーム内文脈）で責務・認可前提が異なるため別エンドポイントにする。

## 8. エッジケース・非機能
- エッジケース:
    - `q` 空／空白のみ → フィルタのみ検索か 422 か（§9）。
    - マッチ 0 件 → 空配列＋ページング情報。
    - `team` に非所属の private チーム指定 → 結果は空（権限的に見えないため）。エラーにするか空にするかは §9。
    - `status=draft`/`in_review`/`archived` 指定だが実行者が著者/Admin でない → 当該記事は除外（フィルタが認可を上書きしない）。
    - 論理削除済み（`deleted_at`）記事は常に対象外。
    - 特殊文字／全文索引の予約語・記号、日本語（トークナイズ）でのマッチ挙動（§9）。
- 認可 / 権限（§8 マトリクス）:
    - `published`＋`public`: Guest 含む全認証ユーザーが結果に含められる。
    - `published`＋`team`: そのチームの Owner/Admin/Member のみ。
    - `draft`/`in_review`/`archived`: 著者と当該チームの Admin/Owner のみ。
    - 認可判定は `team-membership` のロール解決・所属に依存。**検索フィルタは認可を緩めない**（フィルタ ∩ 可視範囲）。
- 失敗時の挙動:
    - 例外はコントローラで `Log::error`＋`report()`＋`ApiResponse::serverError()`（§6 可観測性, `php.md`）。
    - レスポンスは `ApiResponse` エンベロープ統一。ユーザー文言は `lang/en/*.php` 翻訳キー。
- 性能・可用性で意識する点:
    - 検索は**インデックス／全文索引前提**（§6 性能）。`title`/`body` の全文索引、フィルタ列（`team_id`/`status`/`author_user_id`/`deleted_at`）の索引、`article_tags` 経由のタグ絞り込みで N+1 を作らない（Infra で eager load）。
    - 一覧は必ずページング（既定 `Pagination`）。認可絞り込みはできる限りクエリ側で行い、ページング件数の整合を保つ。

## 8.5 テスト方針（準拠先 + 機能固有の観点）
- 準拠 rules:
    - バックエンド（該当する場合）: `.claude/rules/php.md`（PHPUnit 12 / Unit・Feature / 行カバレッジ 100% / `#[Test]` 属性 / Mockery / Feature は MySQL テスト DB）。
    - フロントエンド（該当する場合）: `.claude/rules/typescript.md`（`tsc --noEmit` 型エラーなし・ビルド可、`client` 経由）。
- 本機能固有のテスト観点:
    - 到達が難しい分岐 / 要モックの依存:
        - **検索ポートのモック**: UseCase 単体は `ArticleSearchInterface` を Mockery でモックし、Filter（認可コンテキスト含む）が正しく組まれて渡ること・Output 変換を検証。例外分岐は `andThrow` で UseCase 失敗系、Controller catch は `$this->app->instance(ArticleSearchInterface::class, $throwingMock)` で `serverError` 到達を検証。
        - **認可フィルタの網羅**: ロール（Owner/Admin/Member/Guest）×（`status`×`visibility`）×（所属／非所属 team・public／private）の可視性マトリクス（§8）を Feature で総当たりし、可視範囲外が結果に**含まれない**ことを保証。フィルタ指定が認可を上書きしないことも明示テスト。
    - 重点的に検証したいユースケース（§3 と対応）:
        - US1/US2/US3/US4（キーワード＋各フィルタのマッチ）、US5（認可除外＝最重要）、US6（0 件・ページング）、US7（ポート差し替えでも UseCase 無改修＝バインド差し替えテスト）。
    - DB / 外部サービス依存で注意する点:
        - 全文検索の挙動は MySQL 依存（`MATCH ... AGAINST`/`LIKE`、日本語トークナイズ）。マイグレーションは MySQL 専用生 SQL（`UUID_TO_BIN`/`ENUM`/`COMMENT`/全文索引）で sqlite 不可 → Feature は `learning_portal_testing`/`.env.testing` 前提。
        - 検索品質（日本語・部分一致）の検証は MySQL の全文索引設定（ngram 等）に依存するため、テストデータは実 DB で再現可能な範囲に絞る。

## 9. 未確定（brainstorming で詰める）
- [ ] **MySQL 全文索引 vs `LIKE` の確定**: `title`/`body` を全文索引（`MATCH ... AGAINST`）にするか `LIKE %q%` にするか。両者の併用範囲（タグ語は別扱い等）。タグは正規化済み `tags.name`/`slug` に対する完全/部分一致か。
- [ ] **日本語トークナイズ方針**: MySQL 全文索引の parser（`ngram` か `mecab` プラグインか）。部分一致要件と語境界。`LIKE` 退避時の性能・索引効き。
- [ ] **スコアリング / 並び順**: 関連度スコア（全文検索の relevance）を採用するか、`published_at`/`updated_at` 等の固定ソートか。`sort` クエリの許容値と既定。タイトル一致を本文一致より優先する等の重み付け。
- [ ] **検索ポートの IF 設計**: `ArticleSearchInterface` のメソッドシグネチャ（引数＝Filter/認可コンテキスト、戻り＝Result）。ページング・スコアの受け渡し形。将来 Meilisearch 実装が満たすべき契約（インデックス更新は本ポートの責務外か）。
- [ ] **認可絞り込みの実装位置**: クエリ条件として SQL に落とす（性能優先）か、取得後にドメインで除外するか。可視性ルールを `article-core` と共有する仕組み（共通の認可サービス/Specification を切り出すか、検索側で再実装するか）。
- [ ] **`q` 未指定時の仕様**: 空クエリでフィルタのみ検索を許すか、`q` 必須で 422 か。最小文字数の制約。
- [ ] **フィルタの結合条件**: `tag` 複数指定時の AND/OR、`team`/`status` 複数指定可否。キーワード対象の確定（title+body+tag の重み・対象切替の有無）。
- [ ] **結果のハイライト/抜粋**: マッチ箇所のスニペット生成・ハイライトをサーバで返すか UI 側か。返す場合の本文（Markdown）からの抽出方針。
- [ ] **`team`/`status` に不可視値を指定した場合の応答**: 空配列で返すか 403/422 にするか（情報漏洩防止の観点を含めて整理）。
