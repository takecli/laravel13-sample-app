## 0. メタ
- 機能名: ブックマーク（bookmarks）
- 作成日: 2026-06-27
- grilling 実行: 一部（正本 requirements.md からの転記。未確定は §9）
- 関連機能（依存・関連する他ドラフト）:

    - 認可の前提: `team-membership`（§11「`article-*` は `team-membership`（認可）に依存」。記事可視性＝閲覧可否がブックマーク可否の前提）
    - 類似実装（API 形状の参考）: `article-reactions`（§4.11、P2。`POST/DELETE /articles/{id}/...` のトグル系で構造が近い。ただしブックマークは「ユーザー×記事」で種別なし）
    - リリース段階: **P2**（§4.12 / §11 Phase 2 の 11 番）

## 1. 目的（Why）
Learner が「あとで読む」記事を個人ストックとして登録/解除し、自分のブックマーク一覧から再訪できるようにする（§4.12、US-R2「Learner は記事をブックマークし、あとで読める」、§1.3 提供価値4「ブックマークで知識をストックする」）。リアクション（公開的な評価）と異なり、ブックマークは**本人だけに見える個人的なしおり**である。

## 2. スコープ
- やること:
    - 記事のブックマーク追加（`POST /articles/{id}/bookmark`、§9 API）。
    - 記事のブックマーク解除（`DELETE /articles/{id}/bookmark`、§9 API）。
    - 自分のブックマーク一覧取得（`GET /me/bookmarks`、§9 API）。ページング必須（§6 性能）。
    - データモデル: `bookmarks`（`user_id`×`article_id` 複合 uniq、§5.1）。
- やらないこと（今回対象外）:
    - ブックマークのフォルダ分け・タグ付け・メモ付与（`bookmarks` の列は `user_id`/`article_id` のみ、§5.1。YAGNI §1.4）。
    - 学習パスとしての束ね・順序付け（`collections`〔§4.13、P3〕の役割。ブックマークとは別機能）。
    - 他ユーザーのブックマーク閲覧・共有（ブックマークは本人専用。§13 非ゴール方針に沿う）。
    - 記事一覧 `GET /articles` への「ブックマーク済みか」フラグ埋め込み（必要性は §9 で要確認。本機能の責務外の可能性）。
- 非ゴール（誤解されやすいので明示）:
    - ブックマークは公開的な評価ではない（集計・ランキングへ反映しない。それは `reactions`〔§4.11〕と `gamification`〔§4.16〕の役割）。
    - ブックマークは記事に対してのみ付く（チーム/ユーザー/コレクションへのブックマークは対象外）。

## 3. ユースケース / ユーザーストーリー
- [ ] US1: Learner が閲覧可能な記事をブックマークに追加できる（US-R2、`POST /articles/{id}/bookmark`）。
- [ ] US2: Learner がブックマークを解除できる（`DELETE /articles/{id}/bookmark`）。
- [ ] US3: Learner が自分のブックマーク一覧を新しい順（並び順は §9）にページングで取得し、各記事へ再訪できる（`GET /me/bookmarks`）。
- [ ] US4: 既にブックマーク済みの記事を再度追加しても二重登録されない（`user_id`×`article_id` 複合 uniq、§5.1。冪等）。

## 4. データモデル設計
- エンティティ:
    - `bookmarks`（ユーザー×記事の個人ストック。中間テーブル的だが本機能の主エンティティ）
- 属性 / 型 / 制約（§5.1、§5「全テーブル `id` は `CHAR(36)` UUID、監査列・論理削除方針を踏襲」）:
    - `bookmarks.id`: `CHAR(36)` UUID（PK）
    - `bookmarks.user_id`: FK → `users.id`（ブックマークした本人）
    - `bookmarks.article_id`: FK → `articles.id`（対象記事）
    - 複合 uniq: (`user_id`, `article_id`)（同一ユーザーの同一記事への二重登録を防止、§5.1）
    - 監査列: `created_at` / `updated_at`、作成/更新/削除ユーザーID（§5 既存方針）。`created_at` が「ブックマークした日時」＝一覧の並び順キー候補（§9）。
    - 論理削除（`deleted_at`）を持つか／解除は物理削除かは §9（解除＝レコード削除なので、ブックマークは論理削除より物理削除が自然。要確定）。
- リレーション:
    - `users 1─* bookmarks *─1 articles`（§5.2「articles 1─* … bookmarks」）。
    - enum: ブックマークに区分値 enum は無し（§10 にブックマーク enum の定義なし。reactions の `kind` のような種別は持たない）。

## 5. API 設計
`prefix: /api/v1`、Keycloak セッション必須、`ApiResponse` エンベロープ準拠（§9 前文）。

| メソッド | パス | 概要 | 主なリクエスト | 主なレスポンス | エラー |
|---|---|---|---|---|---|
| GET | `/me/bookmarks` | 自分のブックマーク一覧（記事付き） | クエリ（ページング。§6 一覧はページング必須） | `bookmarks[]`（各 `article` の `id`/`title`/`slug`/著者/`published_at` 等 ＋ `bookmarked_at`） | 401（未認証） |
| POST | `/articles/{id}/bookmark` | 記事をブックマーク追加（冪等） | パス `id`（記事）。ボディなし | 追加後の状態（`bookmarked: true` 等、形は §9） | 401 / 403（記事閲覧権限なし）/ 404（記事なし） |
| DELETE | `/articles/{id}/bookmark` | ブックマーク解除（冪等） | パス `id`（記事）。ボディなし | 解除後の状態（`bookmarked: false` 等、形は §9） | 401 / 404（記事なし。未登録時の扱いは §9） |

- 「自分の」ブックマークは認証ユーザー（`users.id`）で固定。`/users/{id}/bookmarks` のような他者参照は持たない（§9 では `/me/bookmarks` のみ）。
- レスポンスは `{ data, message, result }`（§6 可観測性、CLAUDE.md）。一覧はページング既定（`Pagination`、§6 性能）。

## 6. UI / 画面設計
- 画面 / 状態:
    - 記事詳細の **ブックマークボタン**（トグル: 未登録/登録済みの 2 状態。`article-core` の記事詳細に内包）。
    - **ブックマーク一覧ページ**（`/me/bookmarks`。記事カードのリスト＋ページング。「あとで読む」導線）。
    - 記事一覧カード上のブックマークアイコン（任意。表示するかは「一覧への登録状態埋め込み」要否次第、§9）。
- 主要操作:
    - ブックマーク追加 → `POST /articles/{id}/bookmark`、解除 → `DELETE /articles/{id}/bookmark`。楽観的更新（即座にアイコン切替→失敗時ロールバック）を想定（具体は brainstorming）。
    - 一覧から記事カードクリックで記事詳細へ遷移。
- 入力バリデーション・エラー表示:
    - 入力フォームは無い（パスパラメータのみ）。401 は未ログイン誘導、403 は閲覧不可記事のフィードバック、API 失敗は `ApiError` 捕捉でトグルをロールバック（typescript.md）。

## 7. 技術選定と根拠
- 採用:
    - `bookmarks` テーブル（`user_id`×`article_id` 複合 uniq）による多対多の個人ストック（§5.1）。
    - 追加/解除を **冪等なトグル系 API**（`POST`=登録、`DELETE`=解除）として実装。解除はレコード削除。
    - 一覧は `bookmarks` を起点に `articles` を eager load してページング（§6 性能 N+1 回避）。
- 理由:
    - §5.1 が `bookmarks` を独立テーブル＋複合 uniq で定義済み。複合 uniq により二重登録を構造的に防げる（US4）。
    - reactions（§4.11）と同じ「ユーザー×記事」のトグル構造だが、ブックマークは**種別を持たない**ぶん単純。冪等な POST/DELETE が実装・テストともに分岐が少ない。
- 却下した代替案 / 却下理由:
    - `users` 行に記事ID配列を保持: uniq・索引・一覧ページングが破綻するため却下（§5.1 のテーブル設計に反する）。
    - `collections`（学習パス）でブックマークを兼ねる: 順序付き公開可能なパスと、本人専用のしおりは目的が別（§4.13 vs §4.12）。役割分離のため却下。
    - 単一エンドポイントのトグル（`POST` で状態反転）: 現在状態に依存し冪等性が崩れるため、`POST`/`DELETE` 分離（§9 のエンドポイント定義にも合致）。

## 8. エッジケース・非機能
- エッジケース:
    - 既にブックマーク済みで再度 `POST` → 複合 uniq で二重登録せず成功扱い（冪等、US4）。HTTP ステータス（200 既存 / 201 新規 の区別をするか）は §9。
    - 未登録で `DELETE` → 冪等に成功（204/200）か 404 か、扱いは §9。
    - ブックマーク後に記事が `archived`/非公開化/論理削除された場合、一覧にどう出すか（除外 / グレー表示 / 404 リンク）は §9（**正本に明記なし。要確定**）。
    - `draft`/`in_review` でブックマーク不可（閲覧不可記事はそもそも `POST` で 403、§8）。
    - 記事削除（論理削除）時に紐づく `bookmarks` をどうするか（残す / カスケード解除）は §9。
- 認可 / 権限（§8）:
    - 「コメント・リアクション・ブックマーク」は Owner/Admin/Member ✓、Guest は **public 記事のみ**（§8 マトリクス末尾「（public 記事のみ）」）。
    - ブックマーク可否は**記事の閲覧可否**に従う。`published`＋`team` はメンバー以上、`published`＋`public` は Guest 含め可（§8）。閲覧不可なら `POST` は 403。
    - `GET /me/bookmarks` は本人のもののみ返す（認証ユーザーで強制スコープ。他者のブックマークは参照不可）。
    - 全 API で Keycloak セッション必須（§6 認可）。
- 失敗時の挙動: 例外は `Log::error` ＋ `report()` ＋ `ApiResponse::serverError()` の定型（§6 可観測性、php.md）。
- 性能・可用性で意識する点:
    - `GET /me/bookmarks` はページング必須（§6 性能）。`bookmarks → articles` の結合で N+1 を作らない（Infra で eager load、§6）。
    - `bookmarks` の複合 uniq (`user_id`,`article_id`) に索引。一覧の並び順キー（`created_at` 等、§9）にも索引を検討。
    - ブックマークは同期処理で十分（通知・集計のような非同期化〔§6 可用性〕は不要）。

## 8.5 テスト方針（準拠先 + 機能固有の観点）
- 準拠 rules:
    - バックエンド（該当する場合）: `.claude/rules/php.md`
    - フロントエンド（該当する場合）: `.claude/rules/typescript.md`
- 本機能固有のテスト観点:
    - 到達が難しい分岐 / 要モックの依存:
        - 冪等性分岐: 「未登録→新規作成」と「登録済み→既存維持（二重登録しない）」の双方（Repository インターフェースをモックして UseCase で両分岐到達。複合 uniq の実挙動は Repository の Feature テストで実 DB 検証）。
        - `DELETE` の「登録済み→解除」と「未登録→冪等成功 or 404」分岐（§9 で確定後にテスト）。
        - 認可分岐（メンバー / Guest×public / Guest×team=403）は §8 マトリクス通りに到達させる。
        - 例外定型（catch → `serverError`）は Repository モックの `andThrow` で到達させる。
    - 重点的に検証したいユースケース（§3 と対応）:
        - US1/US2 追加・解除の往復（トグルの冪等性）。
        - US3 一覧が本人のもののみ・新しい順・ページングで返る（他ユーザーのブックマークが混入しないことを必ず検証）。
        - US4 二重 `POST` で行数が増えない（複合 uniq）。
    - DB / 外部サービス依存で注意する点:
        - Feature/Repository テストは MySQL テスト DB 前提（`learning_portal_testing` / `.env.testing`）。マイグレーションは MySQL 専用 SQL（`UUID_TO_BIN` 等）で sqlite 不可（php.md）。
        - `bookmarks` の複合 uniq 制約違反は DB レイヤで起きるため、Repository の Feature テストで実 DB を使って検証する。
