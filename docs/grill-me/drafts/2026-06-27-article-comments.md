> 本ドラフトは `docs/requirements.md`（正本）の確定事項を転記し、未確定のみ §9 に残した。

## 0. メタ
- 機能名: コメント 〔`article-comments`〕
- 作成日: 2026-06-27
- grilling 実行: 一部（正本 requirements.md からの転記。未確定は §9）
- 関連機能（依存・関連する他ドラフト）:
    - **依存（前提）**: `article-core`（§4.6）— コメント対象の `articles`（`article_id`）と可視性/状態。`team-membership`（§4.4）— 認可（ロール×権限、§8）の判定元。`auth`（§4.1）— 認証ユーザー（コメント著者/操作者）。
    - **関連（後続で横断）**: `notifications`（§4.15／P3）— `Notification\Type::comment_added`（§10）。コメント投稿イベントを通知へ乗せるのは notifications 実装後（本機能ではイベント発火点の用意のみ検討、§9）。
    - **リリース順**: MVP の最後（§11 ステップ 8）。`article-core` 完了が前提。

## 1. 目的（Why）
記事に **スレッド型（`parent_id`）コメント** を付け、著者と読者が知識を磨き合えるようにする（§1.3-4, US-C1）。投稿・本人による編集・削除・一覧取得を提供し、記事を起点とした議論・補足・Q&A を蓄積する。MVP のコア価値「書く・読む・育てる」の「育てる」入口にあたる（§1.1）。

## 2. スコープ
- やること:
    - コメント一覧取得（`GET /articles/{id}/comments`）: 対象記事のコメントを **スレッド構造（`parent_id`）** で取得。
    - コメント投稿（`POST /articles/{id}/comments`）: トップレベル or 既存コメントへの返信（`parent_id` 指定）。
    - コメント編集（`PUT /articles/{id}/comments/{cid}`）: **本人のみ**（§4.10）。本文 `body` の更新。
    - コメント削除（`DELETE /articles/{id}/comments/{cid}`）: **論理削除** `deleted_at`（§5 監査列方針）。
    - 認可制御（§8）: コメント可否はロール×記事可視性に従う。
- やらないこと（今回対象外）:
    - リアクション（→ `article-reactions` §4.11／P2）。コメントへのリアクションは正本に定義なし＝対象外。
    - 通知配信そのもの（→ `notifications` §4.15／P3）。`comment_added` 通知の生成・既読管理は別機能。
    - 全文検索でのコメント横断検索（→ `search` §4.14）。本機能の一覧は対象記事内のみ。
    - メンション（@user）・添付・コメントへの添付（正本に定義なし＝対象外）。
- 非ゴール（誤解されやすいので明示）:
    - リアルタイム更新（WebSocket 等）／共同編集（§13）。取得は通常の REST。
    - WYSIWYG リッチ入力。本文は Markdown テキスト前提（記事と同様、§13）。
    - コメントの物理削除（論理削除のみ、§5）。

## 3. ユースケース / ユーザーストーリー
- [ ] US1（US-C1）: 認証ユーザー（記事を閲覧できる権限保持者）は記事にトップレベルコメントを投稿できる。
- [ ] US2（US-C1）: ユーザーは既存コメントに `parent_id` を指定して返信し、スレッドを形成できる。
- [ ] US3: コメント著者は **自分の** コメント本文を編集できる（他者のコメントは編集不可）。
- [ ] US4: コメント著者（および §9 で確定するモデレーション権限保持者）はコメントを削除（論理削除）できる。
- [ ] US5: 記事を閲覧できるユーザーは、その記事のコメント一覧をスレッド構造で取得できる。
- [ ] US6: Guest（未所属認証ユーザー）は **`public` 記事のコメントのみ** 投稿・閲覧でき、`team` 記事では不可（§8）。
- [ ] US7: 論理削除済みコメントは一覧から除外される（または「削除済み」プレースホルダ表示。子コメントの扱いと併せ §9 未確定）。

## 4. データモデル設計
- エンティティ: `comments`（§5.1）。新規 enum なし（区分値なし）。
- 属性 / 型 / 制約（§5 方針 + §5.1 comments 行）:
    - `id` `CHAR(36)` UUID（PK）。
    - `article_id` `CHAR(36)`（FK→`articles`、必須＝コメント対象記事）。
    - `parent_id` `CHAR(36)`（FK→`comments`、**null 可**＝null はトップレベル、非 null は返信先コメント）。
    - `author_user_id` `CHAR(36)`（FK→`users`、必須＝コメント著者）。
    - `body` `Text`（必須。Markdown テキスト。表示時にサニタイズ／§6 XSS）。
    - 監査列: `created_at` / `updated_at`、作成/更新/削除ユーザーID、論理削除 `deleted_at`（§5 既存方針踏襲）。
- リレーション（§5.2）:
    - `comments *─1 articles`（対象記事）。
    - `comments *─1 users(author)`（著者）。
    - `comments 0..1─* comments`（`parent_id` による自己参照＝スレッド）。
- インデックス方針（§6 性能・N+1 回避）: `article_id`（一覧取得）、`parent_id`（スレッド構築）、`deleted_at`（論理削除除外）に索引。一覧は Infra で著者（`users`）を eager load し N+1 を回避。

## 5. API 設計
`prefix: /api/v1`、Keycloak セッション必須。`ApiResponse` エンベロープ（`data/message/result`）準拠（§9, §6）。

| メソッド | パス | 概要 | 主なリクエスト | 主なレスポンス | エラー |
|---|---|---|---|---|---|
| GET | `/articles/{id}/comments` | 記事のコメント一覧（スレッド構造） | path: `id`、クエリ: `page`, `limit`, `sort`（ページング/並びは §9） | `data`: コメント配列（`parent_id` で階層化／フラット＋階層情報は §9） | 401 / 403（記事可視性違反）/ 404（記事不存在）/ 500 |
| POST | `/articles/{id}/comments` | コメント投稿（返信は `parent_id`） | path: `id`、body: `body`, `parent_id`(任意) | 作成されたコメント（201 相当） | 401 / 403（コメント権限なし）/ 404（記事/親不存在）/ 422（本文空・親不正）/ 500 |
| PUT | `/articles/{id}/comments/{cid}` | コメント編集（本人のみ） | path: `id`, `cid`、body: `body` | 更新後のコメント | 401 / 403（本人でない）/ 404（不存在/論理削除済）/ 422 / 500 |
| DELETE | `/articles/{id}/comments/{cid}` | コメント削除（論理削除） | path: `id`, `cid` | 204/成功メッセージ | 401 / 403（権限なし）/ 404 / 500 |

- バリデーションは FormRequest（§6 セキュリティ）。検証済み入力は `toInput()` で Application 層 Input DTO に詰め替え（`php.md` 規約参照）。
- `parent_id` の検証: **同一記事に属するコメントであること**、論理削除済みでないこと、ネスト深さ上限（§9）を FormRequest/UseCase で担保。

## 6. UI / 画面設計
> 記事詳細画面（`article-core`）の下部に統合する想定。フロントは `resources/js`（`typescript.md` 準拠）。

- 画面 / 状態:
    - コメントスレッド（記事詳細内）: トップレベル＋ネストした返信のツリー表示。ローディング/空（「まだコメントがありません」）/エラー状態。
    - コメント投稿フォーム: 本文（Markdown テキストエリア）＋送信。返信時は対象コメント直下にインライン展開。
    - 各コメント: 著者表示名/アバター、投稿日時、編集済み表示、自分のコメントに編集/削除アクション。
- 主要操作:
    - 投稿 / 返信（`parent_id` 付与）/ 編集（本人）/ 削除（確認ダイアログ、論理削除）。
- 入力バリデーション・エラー表示:
    - 本文必須（クライアント側）。サーバ 422 を `ApiError` で受けて表示。
    - 403（権限/可視性）・404 はユーザー向けメッセージへ。`client`（`api/client.ts`）経由・`api/comment.ts` に集約（`typescript.md`）。

## 7. 技術選定と根拠
- 採用:
    - レイヤード/クリーンアーキテクチャの縦切り（`Team`/`article-core` を踏襲）: Input/Output/UseCase/Search(Filter)/Result/Repository interface(`Domains/Repositories`)＋impl(`Infra/Persistence`)/Controller/Request/Resource（§6, `CLAUDE.md`/`php.md`）。
    - スレッドは **隣接リスト（`parent_id` 自己参照）** で保持（§5.1 のモデルに準拠）。階層化（ツリー組み立て）はアプリ層 or 表示層で行う。
    - 本文は Markdown を **プレーンテキストで保存**、レンダリング/サニタイズは表示層（XSS 対策、§6）。
- 理由:
    - 正本 §5.1 が `parent_id` 単一カラムの隣接リストを明示。MVP のスレッドは浅い想定（§9 で深さ上限を確定）で、隣接リストで十分・N+1 は eager load で回避。
    - 編集/削除を「本人」に閉じることで認可がシンプル（§4.10）。モデレーション拡張は §9 で詰める。
- 却下した代替案 / 却下理由:
    - Closure Table / Nested Set（経路列）→ MVP のスレッド要件に対し過剰。YAGNI（§1.4）。隣接リストで足りる。
    - コメントの物理削除 → 監査・スレッド整合（子コメント保持）に反する。論理削除のみ（§5）。
    - コメント専用エンドポイント外出し（`/comments/{cid}` 単独）→ 正本 §9 は記事ネスト（`/articles/{id}/comments/{cid}`）で確定。記事スコープに従う。

## 8. エッジケース・非機能
- エッジケース:
    - `parent_id` が **別記事のコメント** / 不存在 / 論理削除済み → 422 or 404 で拒否。
    - ネストが深すぎる返信（深さ上限超過、§9）→ 422 で拒否、もしくは同一階層へフラット化（方針 §9）。
    - 親コメント削除時の子コメントの扱い（カスケード論理削除 / 残置＋プレースホルダ）→ §9 未確定。
    - 他者コメントの編集/削除試行 → 403。
    - `draft`/`in_review`/`archived` 記事や可視性違反の記事へのコメント → 記事可視性に従い 403/404（記事を閲覧できない＝コメントもできない）。
    - 空本文・空白のみ本文 → 422。
- 認可 / 権限（§8 マトリクス）:
    - **コメント投稿/閲覧**: Owner/Admin/Member（所属チーム記事）。Guest は **`public` 記事のみ**（§8 末尾「（public 記事のみ）」）。
    - **編集**: コメント本人のみ（§4.10）。
    - **削除**: コメント本人（＋モデレーションとして Admin/Owner を含めるかは §9 未確定）。
    - 前提として「その記事を閲覧できること」がコメント操作の必要条件（記事の `status`×`visibility`×ロール、§8 / §7.1）。判定は `team-membership` のロール解決＋`article-core` の可視性に依存。
- 失敗時の挙動:
    - 例外はコントローラで `Log::error`＋`report()`＋`ApiResponse::serverError()`（§6 可観測性, `php.md`）。
    - レスポンスは `ApiResponse` エンベロープ統一。ユーザー文言は `lang/en/*.php` 翻訳キー。
- 性能・可用性で意識する点:
    - 一覧はページング（既定 `Pagination`、§6）。著者を eager load して N+1 回避。`article_id`/`parent_id` に索引。
    - 多数コメント記事でのスレッド組み立てコスト（ページングとツリー化の整合は §9）。

## 8.5 テスト方針（準拠先 + 機能固有の観点）
- 準拠 rules:
    - バックエンド（該当する場合）: `.claude/rules/php.md`（PHPUnit 12 / Unit・Feature / 行カバレッジ 100% / `#[Test]` 属性 / Mockery / Feature は MySQL テスト DB）。
    - フロントエンド（該当する場合）: `.claude/rules/typescript.md`（`tsc --noEmit` 型エラーなし・ビルド可、`client` 経由）。
- 本機能固有のテスト観点:
    - 到達が難しい分岐 / 要モックの依存:
        - 認可 × 記事可視性の **網羅**: ロール（Owner/Admin/Member/Guest）×（記事 `status`×`visibility`）でコメント可否（§8）を Feature で総当たり。特に Guest×public 可 / Guest×team 不可。
        - 編集/削除の **本人判定**: 本人 OK・他者 403・（モデレーション権限が確定すれば）Admin/Owner の削除分岐。
        - `parent_id` バリデーション分岐: 別記事の親・論理削除済み親・不存在親・深さ上限超過の各拒否。
        - 例外分岐到達: Repository interface を Mockery で `andThrow` させ UseCase 失敗系、Controller catch は `$this->app->instance(...)` で `serverError` 到達を検証。
    - 重点的に検証したいユースケース（§3 と対応）:
        - US2（返信スレッド形成）、US3（本人編集）、US4/US7（削除と子コメントの扱い）、US5（スレッド構造取得）、US6（Guest×public/team）。
    - DB / 外部サービス依存で注意する点:
        - マイグレーションは MySQL 専用生 SQL（`UUID_TO_BIN`/`ENUM`/`COMMENT`）。sqlite では動かないため Feature は `learning_portal_testing`/`.env.testing` 前提。
        - `comments` の factory/seeder（親子関係・論理削除済みを含む）を整備。`articles`（`article-core`）依存のフィクスチャが前提。
